# 用途与边界

- 辅助 **Agent + 开发者** 做项目开发：在仓库内维护 **开发契约**（本地 staging），必要时推到 **mdocs**。
- **两个亮点**：① 决策与辅助信息 **落盘**；② 关键结论 **举证**（关键词 + 代码定位，不贴代码正文）。
- 推到 mdocs 时遵循 **`mdocs-cli` skill**（命令、Token、版本锁）。

# 契约目录

**`<项目根>/.mdocs-docs/`**

```
.mdocs-docs/
├── README.md                 # 总索引：文档地图、需求状态
├── map/                      # 机器主索引（全仓，可持续增厚）
│   ├── README.md
│   ├── backend.md
│   ├── frontend.md
│   └── …
├── decisions/
│   └── NNN-<短标题>.md
└── requirements/
    └── <需求名>/
        ├── 需求分析.md       # 人审
        ├── 设计契约.md       # 人审；同意后才开发
        └── 代码索引.md       # 机器；本需求定位点
```

# 原则

| 原则 | 含义 |
|------|------|
| **契约优先** | 先对齐做什么、怎么做、依据在哪，再改代码 |
| **机器要坐标** | `map/`、需求内 `代码索引.md`：关键词 + 路径 + 符号，**不写代码正文** |
| **人要意义** | `需求分析.md`、`设计契约.md` 给人审；业务背景写这里 |
| **设计门控** | **`设计契约.md` 经用户明确同意前，不得开始写业务代码** |
| **本地先行** | 默认只写 `.mdocs-docs/`；用户说「推 mdocs / 落库」才调 CLI |
| **举证** | 结论须指向 map、decisions 或读过的路径；不确定标 `待确认` |

# 读者分工

| 文档 | 主读者 | 内容 |
|------|--------|------|
| `map/*.md` | Agent | 关键词 → 文件/符号 |
| `requirements/.../代码索引.md` | Agent | 本需求相关定位 |
| `requirements/.../需求分析.md` | 人 | 范围、背景、验收 |
| `requirements/.../设计契约.md` | 人（**须审批**） | 方案、变更触达 |
| `decisions/*.md` | 人 + Agent | 为什么这样设计 |

# 场景路由

意图不清时 **只问一句**：新需求、改老需求、还是整理老业务？

| 场景 | 默认动作 | 建哪些文件 |
|------|----------|------------|
| **新需求** | 防重复 → **新建** `requirements/<需求名>/` | 需求分析 → 设计契约 →（同意后）代码索引 |
| **改老需求** | **update** 已有需求夹 | 分析补「本次变更」；改设计契约；**禁止** `xxx-v2` |
| **整理老业务** | **只增厚** `map/` | 不写业务故事；不默认建三件套 |

# 标准流程

```
1. 判场景（新 / 改 / 梳理）
2. 读 .mdocs-docs/README.md、map/、已有需求夹
3. 写或更新 需求分析.md、设计契约.md
4. 呈现设计契约 → 等待用户明确同意
5. ─── 未同意：停止，不得改业务代码 ───
6. 同意后：写代码 + 更新 代码索引.md / map/（若入口变了）
7. 用户要求时：按 mdocs-cli skill 推到 mdocs
```

# map 条目格式

```markdown
### <功能名>

- **关键词**：`keyword1` `keyword2`
- **定位**：
  | 路径 | 符号 | 行号（可选） |
  |------|------|----------------|
  | `src/.../file.ts` | `functionName` | ~40 |
```

- 按 **能力** 分节；符号优先于行号
- 老业务梳理：只往 `map/` 加条目，不写叙述型用户旅程

# map vs 需求内代码索引

| | `map/` | `requirements/.../代码索引.md` |
|--|--------|--------------------------------|
| 范围 | 全仓 | 仅本需求 |
| 篇幅 | 可多文件 | 建议 ≤15 条 |

`设计契约.md` 内 **「变更触达」** = 本次必改文件；代码索引 = 查代码入口。

# 文档模板

### 需求分析.md

```markdown
# <需求名> — 需求分析

> 一句话：本期做什么、不做什么。

## 背景
## 目标
## 非目标（本期不做）
## 验收标准
## 依据
- map：…
- decisions：…
- 待确认：…
```

### 设计契约.md

```markdown
# <需求名> — 设计契约

> **状态**：草案 | 已同意（日期）
> 未经「已同意」不得开始写代码。

## 方案摘要
## 变更触达
| 文件 | 改动说明 |
|------|----------|
## 风险与回滚
## 测试要点
```

### decisions/NNN-标题.md

```markdown
# NNN — 标题

- **状态**：accepted
- **上下文**：…
- **决策**：…
- **后果**：…
```

# 推到 mdocs

仅在用户明确要求时：

1. `search --q "<关键词>"` 查 remote 是否已有
2. 有则 `update <id> --file .mdocs-docs/...`；无则 `create --domain ... --parent ... --file ...`
3. 目录结构尽量与 `.mdocs-docs` 同构
4. 推设计文档前确认状态为「已同意」

# 写代码前自检

- [ ] 已判场景（新 / 改 / 梳理）？
- [ ] `设计契约.md` 是否 **已同意**？（仅改 map 时跳过）
- [ ] 关键结论是否有 map / 路径 / decisions 依据？
- [ ] 改老需求是否 update 原文件夹？
- [ ] 是否未经用户要求就 push mdocs？
